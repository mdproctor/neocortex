# Native Image Configuration Notes

Companion documentation for reachability-metadata.json and native-image.properties.
Each entry explains what it fixes and what error appears without it.

Gate result: **PASS** — ONNX Runtime JNI + DJL Tokenizers JNI both work
in Quarkus native image on macOS ARM (GraalVM 25.0.3, Oracle).

## --initialize-at-run-time entries (native-image.properties)

### ai.onnxruntime (entire package)
All classes in the ai.onnxruntime package deferred to runtime.
OnnxRuntime calls System.loadLibrary() in its static initializer,
but many other classes in the package also have static init that
depends on the native library being loaded. Package-level deferral
ensures the correct initialization order at runtime.
Without (class-level only): UnsatisfiedLinkError on OrtEnvironment.createHandle()
because OnnxRuntime hasn't loaded the native library yet.

### ai.djl.util.Platform
Platform detection and native lib path resolution in static initializer.
Without: build-time init fails — platform detection reads files not available at build time.

### ai.djl.huggingface.tokenizers.jni.LibUtils
Loads the tokenizers native library in static initializer.
Without: build-time init fails — native lib not available at build time.

## reachability-metadata.json — key entries

Generated by GraalVM 25 tracing agent, exercising the full
tokenize → infer → logits pipeline. Critical non-obvious entries:

### ai.djl.huggingface.tokenizers.jni.CharSpan (JNI constructor)
The Rust tokenizer calls back into Java via JNI to create CharSpan
objects (character offset spans in tokenized text). Without this
entry: Rust panics with "called Result::unwrap() on Err(JavaException)".
This was the hardest failure to diagnose — the Rust panic gives no
indication that a missing JNI registration is the cause.

### com.sun.jna.* classes (JNA JNI bridge)
DJL uses JNA (not raw JNI) to load and call the native tokenizer
library. All com.sun.jna classes with jniAccessible=true are
required for the JNA bridge to function in native image. Without
these: DJL fails to load libtokenizers.dylib.

### ai.onnxruntime.OnnxTensor + ai.onnxruntime.TensorInfo (JNI constructors)
ONNX Runtime creates OnnxTensor and TensorInfo objects from native
code via JNI callbacks. Without: inference fails when trying to
return results from session.run().

### Native library resources (resources section)
Both libraries bundle platform-specific native libs inside their JARs.
These must be registered as resources so GraalVM includes them in the
native image and they can be extracted at runtime:
- native/lib/tokenizers.properties — DJL version detection
- native/lib/osx-aarch64/cpu/libtokenizers.dylib — DJL tokenizer native lib
- ai/onnxruntime/native/osx-aarch64/libonnxruntime.dylib — ORT core
- ai/onnxruntime/native/osx-aarch64/libonnxruntime4j_jni.dylib — ORT JNI bridge
- com/sun/jna/darwin-aarch64/libjnidispatch.jnilib — JNA native dispatch

## Discovery process used

1. Manual --initialize-at-run-time for known library loaders
2. GraalVM tracing agent in JVM mode to capture reflection/JNI/resource entries
3. Iterative native build to identify remaining gaps
4. Key insight: @QuarkusMain(name = "native-gate") prevents Quarkus from
   finding the entry point — must use @QuarkusMain without name parameter

## Issues discovered

### @QuarkusMain name parameter breaks command-mode discovery
@QuarkusMain(name = "native-gate") causes Quarkus to not invoke the
QuarkusApplication.run() method. The binary starts and enters idle
mode (no HTTP server, just waiting). Removing the name parameter
fixes it. This may be specific to native image — JVM mode was not
tested with the named variant since we used direct invocation for
JVM tests.

### Package-level vs class-level --initialize-at-run-time for ONNX Runtime
--initialize-at-run-time=ai.onnxruntime.OnnxRuntime alone is insufficient.
OrtEnvironment and other classes have static initialization that depends
on the native library being loaded. Package-level deferral
(--initialize-at-run-time=ai.onnxruntime) is required.
